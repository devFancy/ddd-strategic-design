# 키친포스

## 퀵 스타트

```sh
cd docker
docker compose -p kitchenpos up -d
```

## 요구사항

### 메뉴

- 메뉴를 등록한다.
    - [ ] 메뉴 가격은 0원 이상이어야 한다.
    - [ ] 메뉴는 메뉴 그룹에 속해야 한다.
    - [ ] 메뉴는 1개 이상의 등록된 상품으로 구성된다.
    - [ ] 메뉴 생성 시 지정된 모든 상품이 실제로 등록되어 있어야 한다.
    - [ ] 메뉴 상품 수량은 0개 이상이어야 한다.
    - [ ] 메뉴 가격은 구성 상품의 가격 총 합보다 큰 경우 등록할 수 없다.
    - [ ] 메뉴 이름에는 비속어가 포함될 수 없다
- 메뉴 가격을 변경할 수 있다
    - [ ] 메뉴 가격은 0원 이상 이어야 한다
    - [ ] 메뉴 가격의 합이 구성 상품의 가격 총 합보다 큰 경우 수정할 수 없다.
- 메뉴를 노출 할 수 있다.
    - [ ] 메뉴의 실제 가격이 구성 상품들의 총 가격보다 작거나 같으면 노출 할 수 있다
- 메뉴를 숨길 수 있다.
    - [ ]  메뉴가 존재 하지 않는 경우 숨김 상태로 변경할 수 없다
- [ ] 메뉴를 조회할 수 있다

### 메뉴 그룹

- 메뉴 그룹을 등록한다.
    - [ ] 메뉴 그룹 이름은 빈 값이면 안되고, 필수이다.
- [ ] 메뉴 그룹을 조회할 수 있다.

### 상품

- 상품을 등록 한다.
    - [ ] 상품 가격은 0원 이상 이어야 한다.
    - [ ] 상품 이름에는 비속어가 포함될 수 없다.
- 상품을 가격을 수정 한다.
    - [ ] 상품 가격은 있어야 하고, 0원 이상 이어야 한다.
    - [ ] 메뉴 가격이 상품의 합산 가격 보다 클때 숨길 수 있다.
- [ ] 상품을 조회 한다.

### 주문

- 주문을 생성한다.
    - [ ] 주문은 반드시 주문 유형을 가지고 있어야 한다 (DELIVERY, TAKEOUT, EAT_IN).
    - [ ] 주문은 한 개 이상의 주문 상품을 가지고 있어야 한다.
    - [ ] 주문 항목의 메뉴는 실제로 존재해야 한다.
    - [ ] 매장 식사가 아닌 경우에는 수량이 음수일 수 없다.
        - 매장식사의 경우는 음수수량을 허용한다.
    - [ ] 메뉴가 존재하고, 메뉴가 노출 되어있는 경우에만 주문할 수 있다.
    - [ ] 요청된 주문 가격이 메뉴 가격과 일치해야 한다.
    - 배달 주문
        - [ ] 배달 주문의 경우 주소가 필수이다.
    - 매장 주문
        - [ ] 매장 주문의 경우 테이블 번호가 필수이다.
        - [ ] 테이블을 고객이 사용중 이어야 한다.
    -
- 주문 상태 관리
    - 주문을 접수한다.
        - [ ] 대기(WAITING) 상태의 주문만 접수할 수 있다.
        - [ ] 배달 주문(DELIVERY)의 경우 배달 서비스에 배달 요청을 한다.
        - [ ] 주문 접수 시 주문 상태는 접수(ACCEPTED)로 변경된다.
    - 주문을 서빙한다.
        - [ ] 주문 상태가 접수(ACCEPTED)인 주문만 서빙할 수 있다.
        - [ ] 주문 서빙 시 주문 상태는 서빙(SERVED)로 변경된다.
    - 주문을 배달한다.
    -
        - [ ] 주문 상태가 서빙(SERVED)인 주문만 배달할 수 있다.
    - [ ] 주문 타입이, 배달(DELIVERY)인 주문만 배달할 수 있다.
    - [ ] 주문 배달 시 주문 상태는 배달(DELIVERING)로 변경된다.
    - 배달을 완료한다.
        - [ ] 주문 상태가 배달 중(DELIVERING)인 주문만 배달 완료(DELIVERED)할 수 있다.
    - 주문을 완료한다.
        - [ ] 주문 타입이 배달(DELIVERY)인 경우, 주문 상태가 배달 완료(DELIVERED)된 주문만 완료할 수 있다.
        - [ ] 주문 타입이 매장식사(EAT_IN)와 테이크아웃(TAKEOUT)인 경우 주문 상태가 서빙 완료(SERVED)인 경우만 완료할 수 있다.
        - [ ] 매장식사 주문이 완료되면 해당 테이블의 주문이 모두 완료된 경우 테이블을 초기화한다
- [ ] 주문을 조회한다.

### 매장 테이블

- 매장 테이블을 등록한다.
    - [ ] 테이블 이름은 빈 값이면 안되고, 필 수이다.
    - [ ] 초기 손님 수는 0명으로 설정 하고, 매장 테이블 상태는 비어있음으로 설정한다.
- 매장 테이블 관리
    - 매장 테이블 상태를 비어있지 않음으로 변경한다.
        - [ ] 매장 테이블이 존재해야 한다.
    - 매장 테이블 상태를 비어있음으로 변경한다.
        - [ ] 매장 테이블이 존재해야 한다.
        - [ ] 주문 상태가 완료(COMPLETED)인 매장 테이블만 비어있음으로 변경할 수 있다.
    - 매장 테이블의 손님수 변경
        - [ ] 매장 테이블이 존재 해야 한다.
        - [ ] 매장 테이블의 손님 수는 0명 이상 이어야 한다.
        - [ ] 사용 중인 테이블의 손님 수만 변경할 수 있다/
- [ ] 매장 테이블을 조회 한다.

## Kitchen POS System 모델링

레스토랑의 주문과 메뉴를 관리하는 POS 시스템입니다.

### 메뉴

#### 속성

- `메뉴`는 `이름`, `가격`, `노출 여부`를 가지고 있다.
- `메뉴`는 `메뉴 그룹`에 속해야 한다.
- `메뉴`는 1개 이상의 `상품`으로 구성 된다.

#### 행위

- `메뉴`를 등록 정책에 따라 등록한다.
- `메뉴`를 노출 시킨다.
- `메뉴`를 숨긴다.
- `메뉴`의 가격을 변경한다.
- `메뉴`를 조회한다.

#### 정책

- `메뉴`의 가격은 0원 이상 이어야 한다.
- `메뉴`는 `메뉴 그룹`에 속한다.
- `메뉴`는 1개 이상의 등록된 `상품`으로 구성 되어야 한다.
- `메뉴` 생성 시 가격 정책을 만족 시켜야 한다.
    - `메뉴`의 가격은 구성 `상품`의 가격 총 합보다 큰 경우 등록 할 수 없다.
- `메뉴` 노출 시킬시 가격 정책을 만족 시켜야 한다.
    - `메뉴`의 실제 가격이 구성 `상품`들의 총 가격보다 작거나 같으면 노출 할 수 있다.
- `메뉴`의 이름에는 비속어가 포함될 수 없다.
    - `비속어` 외부 시스템을 통해 비속어가 포함되어 있는지 확인한다.

### 메뉴 그룹

#### 속성

- `메뉴 그룹`은 `메뉴`를 분류하는 카테고리이다.
    - 1:N 관계로 `메뉴들`은 하나의 `메뉴 그룹`에 속한다.
    - ex. 덮밥, 세트 요리, 음료 등
- `메뉴 그룹`은 `이름`을 가지고 있다.

#### 행위

- `메뉴 그룹`을 등록한다.
- `메뉴 그룹`을 조회한다.

#### 정책

- `메뉴 그룹`의 이름은 빈 값이면 안되고 필수이다.

### 상품

#### 속성

- `상품`은 `이름`, `가격`을 가지고 있다.
- `상품`은 `메뉴`를 구성하는 최소 단위이다.

#### 행위

- `상품`을 등록한다.
- `상품`의 가격을 변경할 수 있다.
- `상품`을 조회할 수 있다.

#### 정책

- `상품`의 가격은 0원 이상이어야 한다.
- `상품`의 이름에는 비속어가 포함될 수 없다.
- `상품`가격에 따라 상품이 속해 있는 메뉴가 숨겨지게 될 수 있다.

### 주문 공통

#### 속성

- `주문`은 `주문 유형`, `주문상태`, `주문 상품 목록`을 가지고 있다.

#### 정책

- `주문`은 반드시 `주문 유형`을 가지고 있어야 한다.
    - `주문 유형`은 배달, 포장, 매장식사 중 하나여야 한다.
- `주문`은 한개 이상의 `메뉴`를 갖는다.
- `주문`은 `메뉴`가 존재 해야 하고, 노출되어 있어야 한다.
- `주문`은 `주문 가격`이 `메뉴 가격`과 일치해야 한다.

### 배달 주문

#### 속성

- `배달 주문`은 `배달 주소`, `배달 라이더 대행사`, `배달 주문 상태`, `배달 주문 접수 시각`을 가지고 있다.
- `주문 유형`은 `배달 주문`이다.

#### 행위

- `배달 주문`을 주문한다.
- `배달 주문`을 접수한다.
- 고객이 `접수`한 `배달 주문`을 `라이더`에 전달한다.
- `라이더`가 메뉴를 배달한다.
- `라이더`가 배달을 완료한다.
- `배달 주문`을 완료한다.

#### 정책

- `배달 주문`은 `배달 접수`시 `라이더`에게 배달 요청을 한다.
- `배달 주문`은 `배달 주소`를 가지고 있어야 한다.
- `주문항목`은 적어도 한개 이상 있어야 한다.
- `배달 주문`은 `배달 주문 상태`를 가지고 있다.
- `배달 주문`은 `배달 주문 상태`가 `배달 완료`인 경우에만 완료할 수 있다.
    - `배달 완료` 상태는 `라이더`가 고객에게 배달을 완료한 상태이다.
- `배달 주문`은 주문 상태가 총 6개 있다.
    - `접수 대기`, `접수`, `배달 준비 완료`, `배달 중`, `배달 완료`, `주문 완료`

### 매장 식사 주문

#### 속성

- `매장 식사 주문`은 `테이블 번호`, `매장 주문 상태`, ` 매장 주문 접수 시각 `를 갖고 있다.
- `주문 유형`은 `매장 식사 주문`이다.

#### 행위

- `매장 식사 주문`을 주문한다.
- `매장 식사 주문`을 접수한다.
- `매장 식사 주문`을 서빙한다.
- `매장 식사 주문`을 완료한다.
    - `매장 테이블`을 `비워있음`으로 변경하려면 `매장 식사 주문`이 완료되어야 한다.

#### 정책

- `매장 식사 주문`은 `테이블 번호`를 가지고 있어야 한다.
- `매장 식사 주문`은 `매장 테이블`을 고객이 사용중이어야 한다.
- `매장 식사 주문`은 `매장 주문 상태`를 가지고 있어야 한다.
- `매장 식사 주문`은 `매장 주문 상태`가 `서빙 완료`인 경우에만 `주문 완료`할 수 있다.
- `매장 식사 주문`은 `매장 주문 상태`가 `주문 완료`인 경우에만 완료한다.
    - `주문 완료` 상태는 `고객`이 식사를 완료한 상태여야 한다.
    - 위 과정이 끝나면 `매장 테이블`을 비울 수 있다.
- `매장 식사 주문`이 완료되면, 해당 `매장 테이블`을 비워야 한다.
- `매장 식사 주문`은 주문 상태가 총 4개 있다.
    - `접수 대기`, `접수`, `서빙 완료`, `주문 완료`

### 포장 주문

#### 속성

- `포장 주문`은 `포장 주문 상태`를 가지고 있다.
- `주문 유형`은 `포장 주문`이다.

#### 행위

- `포장 주문`을 주문한다.
- `포장 주문`을 접수한다.
- `포장 주문`의 준비를 완료하여 고객에게 전달한다.
- `포장 주문`을 완료한다.

#### 정책

- `포장 주문`은 `포장 주문 상태`를 가지고 있다.
- `포장 주문`은 `포장 주문 상태`가 `서빙 완료`인 경우에만 고객에게 인계할 수 있다.
    - `서빙 완료` 상태는 `고객`이 포장 주문을 받아가는 것을 말한다.
- `포장 주문`은 주문 상태가 총 4개 있다.
    - `접수 대기`, `접수`, `서빙 완료`, `주문 완료`

### 매장 테이블

#### 속성

- `매장 테이블`은 `매장 테이블 번호`, `매장 테이블 이름`, `매장 테이블 방문 손님 수`, `매장 테이블 여부`를 가지고 있다.

#### 행위

- `매장 테이블`을 등록한다.
- `매장 테이블`을 조회한다.
- `매장 테이블`의 손님 수를 변경한다.
- `매장 테이블`을 사용중으로 변경한다.
- `매장 테이블`을 비어있음으로 변경한다.

#### 정책

- `매장 테이블`은 `매장 테이블 번호`를 가지고 있어야 한다.
- `매장 테이블`은 `매장 테이블 이름`을 가지고 있어야 한다.
    - `매장 테이블 이름`은 빈 값이면 안되고, 필수이다.
- `매장 테이블`은 `매장 테이블 방문 손님 수`를 가지고 있다.
- `매장 테이블`은 `매장 테이블 사용 중`인 경우에만 손님 수를 변경한다.
- `매장 테이블`은  `주문 상태`가 `완료`인 경우에만 비어있음으로 변경한다.

### 외부 시스템

- `라이더 대행사`는 라이더를 통해서 배달 주문을 처리한다.
- `비속어 검증`은 메뉴 이름에 비속어가 포함되어 있는지 확인한다.

### 사용자

- `고객`은 `주문`을 한다..
- `업주`는 `주문`을 접수하고, `메뉴`를 관리한다.
- `라이더`는 `배달 주문`을 배달한다.

### 주문 프로세스 플로우

```mermaid
graph TD
    subgraph Customer["고객 Flow"]
        A[고객] --> B{주문 유형 선택}
        B -->|배달 주문 flow| C1[배달 주문]
        B -->|포장 주문 flow| C2[포장 주문]
        B -->|매장 식사 flow| C3[테이블 착석]
        C3 --> C4[매장 주문]
        C1 --> D[메뉴 선택]
        C2 --> D
        C4 --> D
        D --> E[주문 요청]
    end

    subgraph Owner["업주 Flow"]
        G[주문 접수] --> H{주문 타입 확인}
        H -->|배달 주문| J1[주문 수락]
        H -->|포장 주문| J2[주문 수락]
        H -->|매장식사| J3[주문 수락]
        J1 & J2 & J3 --> K[메뉴 준비]
        K -->|배달 주문| L1[라이더 픽업]
        L1 --> M1[배달 완료]
        K -->|포장 주문| L2[포장 완료]
        L2 --> M2[고객 수령]
        K -->|매장식사| L3[서빙]
        L3 --> M3[식사 완료]
    end

    subgraph Rider["라이더 대행사 가상 Flow"]
        N[대행사] --> O[라이더 배정]
        O --> P[라이더 가게 도착]
        P --> Q[라이더 음식 수령]
        Q --> R[라이더 고객에게 배달 완료]
    end

    E -->|주문 전송| G
    J1 -->|배달 요청| N
    R -->|배달 완료 처리| M1
```
